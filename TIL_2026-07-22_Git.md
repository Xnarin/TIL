# 1. Git & GitHub

## 1.1 버전 관리와 Git

### 버전 관리가 필요한 이유

버전 관리는 프로젝트의 변경 이력을 관리하는 방법이다.

버전 관리를 통해 다음과 같은 장점을 얻을 수 있다.

- 이전 상태로 복구 가능
- 여러 사람과 협업 가능
- 데이터 손실 방지 및 백업 가능

### Git을 사용하는 이유

Git은 분산 버전 관리 시스템(DVCS, Distributed Version Control System)이다.

각 사용자가 저장소 전체를 가지고 있기 때문에 중앙 서버에 문제가 발생해도 작업 내용을 유지할 수 있다.

또한 여러 개발자가 동시에 작업하고 변경 사항을 효율적으로 관리할 수 있다.


<br>


# 2. Git을 이용한 버전 관리

## 2.1 Commit 실습

Git 저장소 생성부터 GitHub 업로드까지의 과정

```bash
git init

파일 작성 및 수정

git add .

git commit -m "커밋 메시지"

git remote add origin <원격저장소URL>

git push
```

각 명령어의 역할:

| 명령어 | 설명 |
| --- | --- |
| `git init` | 현재 폴더를 Git 저장소로 초기화 |
| `git add` | 변경 사항을 Staging Area에 추가 |
| `git commit` | 변경 사항을 버전으로 저장 |
| `git push` | Commit 내용을 원격 저장소(GitHub)에 업로드 |


<br>

# 3. Git 저장 구조

Git은 파일의 변경 사항을 다음 단계로 관리한다.

```
Working Directory
        |
        | git add
        ↓
Staging Area
        |
        | git commit
        ↓
Local Repository (Commits)
        |
        | git push
        ↓
Remote Repository (GitHub)
```

## Working Directory

실제로 파일을 생성하고 수정하는 프로젝트 폴더이다.

코드를 작성하거나 파일을 변경하는 공간이다.



## Staging Area

다음 Commit에 포함할 변경 사항을 선택해 두는 공간이다.

`git add` 명령어를 실행하면 변경 사항이 Staging Area에 등록된다.

Staging Area를 사용하는 이유:

- Commit 전에 변경 사항 검토 가능
- 원하는 파일만 선택해서 Commit 가능
- 변경 내용을 의미 있는 단위로 나누어 관리 가능



## Commit

변경 사항을 하나의 버전으로 확정하고 저장하는 공간이다.

각 Commit에는 고유한 해시 값이 부여된다.

Commit을 통해 프로젝트의 특정 시점으로 돌아갈 수 있다.

# 4. Git 명령어 정리

| 명령어 | 설명 |
| ------ | ---- |
| `git init` | 현재 폴더를 Git 저장소로 초기화 |
| `git status` | 현재 파일 변경 상태 확인 |
| `git add .` | 변경된 모든 파일을 Staging Area에 추가 |
| `git commit -m "메시지"` | 변경 내용을 Commit으로 저장 |
| `git log --oneline` | Commit 기록을 한 줄로 확인 |
| `git config --global user.name "이름"` | Git 사용자 이름 설정 |
| `git config --global user.email "이메일"` | Git 사용자 이메일 설정 |
| `git remote add origin <원격저장소URL>` | 원격 저장소 연결 |
| `git remote -v` | 연결된 원격 저장소 확인 |
| `git push origin master` | Local Repository의 Commit을 원격 저장소로 업로드 |   



<br>


# 5. Git 질문 정리

## `git push origin master` 와 `git push -u origin master` 차이

### git push origin master

```bash
git push origin master
```

현재 브랜치의 내용을 `origin` 저장소의 `master` 브랜치로 Push한다.

하지만 이후 Push할 때마다 원격 저장소와 브랜치를 직접 지정해야 한다.



### git push -u origin master

```bash
git push -u origin master
```

`-u` 옵션은 업스트림(upstream)을 설정한다.

현재 Local Branch와 Remote Branch를 연결한다.

한 번 설정하면 이후부터는:

```bash
git push
```

만 입력해도 자동으로 연결된 원격 저장소로 Push된다.

# 6. CLI(Command Line Interface)

## GUI (Graphic User Interface)

그래픽 화면을 통해 사용자와 컴퓨터가 상호작용하는 방식이다.

예:

- 파일 탐색기
- VS Code 화면



## CLI (Command Line Interface)

터미널에서 명령어를 입력하여 컴퓨터와 상호작용하는 방식이다.

예:

```bash
git status

python main.py
```



## CLI를 사용하는 이유

- 적은 시스템 자원 사용
- 세부적인 기능 제어 가능
- 반복 작업 자동화 가능
- 서버 및 개발 환경에서 많이 사용

많은 서버와 개발 시스템은 CLI 환경을 기반으로 동작한다.

# 7. Git 저장소 복제하기

## Repository Clone

GitHub 저장소를 Local 환경으로 복제한다.

```bash
git clone <https://github.com/사용자명/git-practice.git>
```



## 복제된 폴더 이동

```bash
cd git-practice
```

Clone한 저장소 폴더로 이동한다.



## VS Code 실행

```bash
code .
```

현재 폴더를 VS Code에서 연다.

만약 `code .` 명령어가 동작하지 않는 경우:

```
VS Code 실행
→ 파일
→ 폴더 열기
→ git-practice 선택
```

# 8. 정리

### Git의 기본 흐름

```
파일 작성 및 수정
        ↓
Working Directory
        ↓
git add
        ↓
Staging Area
        ↓
git commit
        ↓
Local Repository
        ↓
git push
        ↓
GitHub Remote Repository
```

### 용어 정리

- **Stage** : 이번 Commit에 포함할 변경 사항을 선택하는 공간
- **Commit** : 선택한 변경 사항을 하나의 버전으로 저장
- **Push** : Commit을 GitHub 원격 저장소로 전송
